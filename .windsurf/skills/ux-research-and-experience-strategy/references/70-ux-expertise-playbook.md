# UX Expertise Playbook for Vague Prompts

Use this playbook when prompts are broad (for example "improve UX") but enough context exists to proceed safely.

## Scope-Safe Default Output

Deliver this baseline package first:

1. User problem framing (who, pain point, desired behavior change).
2. Decision-focused research plan (method, sample, confidence target).
3. Prioritized friction list (severity + expected impact).
4. Proposed UX improvements tied to specific journeys.
5. CTA clarity review (primary action understanding, ambiguity, conflict risk).
6. Validation plan (tasks, metrics, pass/fail criteria).
7. Rollout and learning loop (instrumentation + follow-up checks).

## Evidence Quality Baseline

- Triangulate at least two evidence streams when possible (research + analytics, or research + support signals).
- Label confidence level and key unknowns.
- Avoid sweeping recommendations when evidence confidence is low.
- Ask concise clarifying questions only for blocking ambiguities.

## Journey and Consistency Baseline

- Check continuity across entry, core task, error recovery, and follow-up states.
- Keep terminology/actions consistent across related flows.
- Ensure proposed improvements do not optimize one screen while degrading adjacent workflows.

## CTA Clarity Baseline

- Ensure each step has a clear primary next action.
- Avoid conflicting CTA choices that increase hesitation.
- Validate whether users can predict CTA outcomes before clicking.
- Check destructive/irreversible CTAs for confirmation and recovery patterns.
- Check CTA placement consistency across steps so users can form reliable expectations.
- Flag CTA color misuse when visual priority does not match action priority or risk level.

## Theme-Mode Usability Baseline

- Verify critical journeys in both light and dark modes when both are supported.
- Check text readability and hierarchy continuity across mode switching.
- Check CTA/button confidence in both modes (state clarity, affordance, and semantic color meaning).
- Flag mode-specific regressions where one mode degrades decision clarity.

## Named Product-Family Baseline

- If the request references an existing product family, research the live product family before recommending change.
- Preserve the familiar journey skeleton unless evidence shows it is the source of user harm.
- For messaging products, define how users scan recent conversations, recover the correct thread, send confidently, and recover drafts or failed sends before redesigning surface details.

## Noise Reduction Baseline

- Reduce unnecessary options shown at each decision point.
- Remove repeated instructional text that does not change user action.
- Keep critical task paths short and free of distracting side actions.

## Prioritization Baseline

- Prioritize by user impact, confidence, effort, and risk.
- Separate "must-fix now" from "iteration candidate" items.
- Keep recommendations actionable and testable by engineering/product teams.

## Default Clarifying Questions (Only if Blocking)

1. Which user segment or job is highest priority?
2. Which task/flow is most business-critical right now?
3. What constraints apply (timeline, compliance, accessibility)?
4. Do we have existing data/research to reuse before new discovery work?
